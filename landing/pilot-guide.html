Part 1: Executive Overview

What Is Chimera Secured?

Chimera Secured is a behavioral email security platform that detects Business Email Compromise (BEC) by analyzing how each person writes, not what they say. It builds a multi-dimensional behavioral fingerprint for every protected mailbox and flags outbound emails that deviate from the owner's unconscious writing patterns — even when the email comes from the real, authenticated account.

Unlike traditional email security tools that rely on known threat signatures, reputation scoring, or content-keyword rules, Chimera Secured catches attacks that use legitimate credentials and clean language — the exact scenario that BEC exploits.

Why MSPs Should Offer This

  • BEC is the #1 cybercrime by dollar loss: $3.0B reported to FBI IC3 in 2025 alone.
  • Traditional SEGs and DMARC don't catch account-takeover BEC — the attacker is inside the mailbox.
  • Runs inside the customer's M365 tenant. Email bodies never leave their environment, eliminating the most common objection to third-party email scanning.
  • Recurring revenue model: Per-mailbox monthly licensing with tiered pricing for MSP partners.
  • Genuine differentiator: No competing product uses behavioral stylometry at this depth.

Pilot Success Criteria

A pilot is considered successful when the following thresholds are met:

  • False-positive rate < 2% across all monitored mailboxes.
  • Detection demonstrated on at least one simulated BEC scenario.
  • Behavioral profiles successfully built for 80%+ of enrolled mailboxes.
  • Customer security team confirms dashboard visibility and alert quality.
  • No email bodies observed leaving the customer's M365 tenant boundary.

Part 2: Setup & Deployment

Prerequisites

Customer Environment

  • Microsoft 365 tenant (Business Premium, E3, or E5).
  • Exchange Online mailboxes (not on-prem Exchange).
  • Global Admin or Exchange Admin credentials for initial consent.
  • Azure AD app registration permissions.
  • Docker or container runtime in tenant infrastructure (ACI, AKS, or equivalent).

  • Access to the Chimera Secured Partner Portal for license keys and container images.
  • Familiarity with M365 Admin Center and Azure AD app registrations.
  • Designated pilot lead on the MSP side (single point of contact).

Step-by-Step Deployment

Step 1: Register the Azure AD Application

Chimera Secured requires an Azure AD app registration to read outbound email history via the Microsoft Graph API.

PermissionTypePurpose
Mail.ReadApplicationRead sent-mail history for behavioral profile enrollment and ongoing outbound email scoring.

Important: Only the Mail.Read application-level permission is required, admin-consented by a Global Admin. The system reads sent-mail history during enrollment and monitors outbound mail flow for scoring. Email bodies are processed in memory and never persisted or transmitted outside the tenant.

Step 2: Deploy the Analysis Container

  1. Log in to the Chimera Secured Partner Portal and generate a deployment token.
  2. Pull the container image using the provided registry credentials.
  3. Deploy to Azure Container Instances (ACI) or AKS within the customer's subscription.
  4. Configure the required environment variables (see Step 3).
  5. Verify the container starts and the /health endpoint returns 200.

Step 4: Enroll Initial Mailboxes

Start with a targeted set of high-value mailboxes for the pilot:

  • C-suite executives (CEO, CFO, COO) — highest BEC target value.
  • Finance team leads — wire-transfer authority.
  • HR leadership — payroll and PII access.
  • IT administrators — credential and infrastructure access.

Recommended pilot size: 5–10 mailboxes. Focused enough for hands-on management during the pilot, with enough diversity to validate detection across different writing styles.

Enrollment is triggered per-mailbox via the admin dashboard or API. The system reads the user's sent-mail history, builds a multi-dimensional behavioral profile, then discards the raw email content. Only the abstract behavioral fingerprint is retained. Enrollment takes 30–120 seconds per mailbox.

Step 5: Verify Shadow Mode

  1. Open the admin dashboard and confirm all enrolled mailboxes show "Profile Active" status.
  2. Send a test email from an enrolled mailbox and confirm it appears in the scoring log.
  3. Confirm that no mail-flow disruption has occurred — Chimera operates in read-only mode.
  4. Review the /health endpoint: verify db_healthy=true and all services report ready.

Part 3: Managing the Pilot

Dashboard Overview

Detection Feed

A real-time stream of scored outbound emails. Each entry shows: sender, recipient, authenticity score (0–1.0), confidence level, which analysis layer(s) fired, and the content-risk classification.

Profile Health

Per-mailbox view showing: profile version, training email count, coverage across formality contexts (formal/standard/casual), last-updated timestamp, and any warnings.

Relationship Labels

The auto-classification system assigns each recipient a formality context. The labeling queue lets the MSP or customer security team review and override these assignments to improve detection accuracy.

Part 4: Testing & Validation

A pilot that only runs in shadow mode and waits for a real attack proves nothing. Chimera Secured provides structured testing scenarios that demonstrate detection capability using controlled, safe simulations.

Test 1: Baseline Verification

Purpose: Confirm that legitimate outbound emails from enrolled users score high (> 0.7) with good confidence.

  1. Select 5 enrolled mailboxes with "Profile Active" status.
  2. Have each user send 3 normal business emails to different recipients.
  3. Wait 5 minutes for scoring to process.
  4. Open the Detection Feed and filter by sender.
  5. Verify each email scored > 0.7 with confidence > 0.5.

Expected: All 15 emails score as authentic. If any score below 0.7, check profile health and relationship labels.

Test 2: Style-Deviation Detection

Purpose: Confirm that emails written by a different person from the same mailbox are flagged.

  1. Select 3 enrolled mailboxes (with owner's permission).
  2. Have someone other than the mailbox owner compose and send 2 emails from each account. The stand-in should write naturally — do not mimic the owner.
  3. Wait 5 minutes for scoring.
  4. Verify stand-in emails score significantly lower (typically < 0.4).

Expected: Clear separation between owner-written and stand-in-written emails. This demonstrates the core detection capability.

Test 3: BEC Simulation

Purpose: Simulate a realistic BEC attack using Chimera Secured's built-in test harness.

  1. Access the Test Console from the admin dashboard.
  2. Select target mailbox and attack tier.
  3. The test harness generates a simulated BEC email through the scoring pipeline. No actual email is sent.
  4. Review the score and analysis-layer breakdown.
Attack TierDescriptionExpected Result
Tier 1: GenericStandard BEC template. No personalization.Strong detection — high confidence flag
Tier 2: Context-AwareUses target's role, company, recent activity.Reliable detection — medium-high confidence flag
Tier 3: Style-MimickingAI-generated content mimicking target's style.Detection with lower confidence — multiple layers contribute

Test 4: False-Positive Stress Test

Purpose: Confirm that edge-case legitimate emails are not flagged.

Test these scenarios from enrolled mailboxes:

  • An email written in an unusual emotional state (formal user sends casual congrats).
  • A very short email (< 15 words, e.g., "Approved. Please proceed.").
  • An email with heavy quoting/forwarding.
  • An email to a brand-new recipient not in training history.

Expected: Short emails return lower confidence (system knows it has less signal). Emotional-shift emails may score slightly lower but remain above alert threshold. New-recipient emails use fallback formality context and should still score > 0.6.

API Reference

All endpoints run inside the customer's tenant boundary. The analysis engine runs on port 8400; the scoring service runs on port 8500.

EndpointMethodPurpose
/healthGETContainer and database health check.
/enrollPOSTBuild or refresh a behavioral profile for one user.
/scorePOSTScore one email against a user's behavioral profile. Returns p_authentic (0–1) and confidence (0–1).
/profile-statusGETQuery a user's behavioral profile metadata.
/labeling-queueGETRetrieve recipient classification queue for review.
/labelPOSTSubmit or override a formality-context label.
/labeling-progressGETSummary stats on labeling completion.
